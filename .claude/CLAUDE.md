## 절대 원칙 (지시서 2번)
1. **AI는 거울이지 분석가·선생·상담사가 아니다.** 채점하지 않고, 정답을 주지 않고, 진단하지 않고, 사용자를 유형이나 성격으로 명명하지 않는다. AI의 출력은 항상 "대조"와 "회수"다.
2. **사용자가 쓰지 않은 것을 지어내지 않는다.** AI가 과거 기록을 참조할 때는 반드시 원문 인용 + 작성 날짜 형태로만 한다. 요약·의역·패턴 명명 금지. 인용할 문장이 없으면 인용하지 않는다.
3. **톤 규칙:** 위로 언어 금지. 이모지 금지. 감탄·과잉 반응 금지. 극적인 비유 금지. 판단·충고·조언·평가 금지. 영어 외래어 최소화. 짧은 단정문. ~이에요/~해요체. 건조하고 중립적인 거울의 말투.
4. **추상 언어의 구체화는 바꿔주기가 아니라 되묻기로 한다.** 사용자가 "힘들었다"고 쓰면 AI가 "지쳤다"로 바꾸지 않는다. "'힘들었다'고 쓰셨어요. 오늘 어떤 장면에서였나요"처럼 사용자가 스스로 구체화할 자리를 만든다.
5. **행동의 주어는 사용자다.** 하루의 마지막 단계에서 작은 행동을 정하는 것은 사용자이며, AI 제안은 사용자가 명시적으로 요청할 때만 노출한다.
6. **사용자 간 데이터 격리는 브랜드 생사 문제다.** 격리(데이터베이스 층)와 공유(제품 층)를 구분한다. 격리는 예외 없이 적용하고, 공유는 사용자의 건별 선택으로만 일어난다.

## 관점 원칙 (사용자 지시, 2026-07-05 — 전 화면·문구·설문에 적용)
- 오제로는 사용자를 문제 있는 사람·고쳐야 하는 사람·아픈 사람으로 보지 않는다. **환자·내담자가 아니라 관찰자, 무엇이든 해낼 수 있는 사람**으로 대한다.
- 따라서 화면 문구·안내문·알림에 결핍·치료 어휘 금지: 증상·치료·회복·개선·문제·교정·극복·힐링 등. 측정·설문 도구를 고를 때도 결핍 진술형("생각이 나를 괴롭힌다")보다 능력 진술형("나는 생각을 생각으로 볼 수 있다")을 우선한다.

## 판독 규칙 — 사실·망상 판별 기준 (사용자 지시, 2026-07-05 — AI 거울 대조(6단계) 포함 모든 AI 동작의 공통 기준)

**정의**
- 사실 = 그 자리에 카메라와 녹음기가 있었다면 찍히고 녹음되었을 것. 일어난 일, 관찰 가능한 행동, 실제로 발화된 문장, 숫자, 시각.
- 망상 = 그 위에 마음이 붙인 모든 것. 해석, 추측, 의미 부여, 예측, 평가, 일반화. 무의식의 패턴이 자동으로 생산한 해석이며 "틀린 생각·나쁜 생각"이 아니다. 제거 대상이 아니라 무의식 상태를 보여주는 단서다.
- 용어 규칙: 이 앱은 "해석" 대신 **"망상"이라는 단어를 의도적으로 쓴다. 다른 단어로 순화하지 마라.**

**판별 검사 두 개 (순서대로)**
1. 카메라 검사: 카메라에 찍히는가? — "그가 인사 없이 지나갔다" 찍힘=사실 / "그가 나를 무시했다" 무시는 안 찍힘=망상
2. 근거 검사 (카메라 검사가 애매하면): 이 문장의 근거가 관찰인가 추론인가? 추론이면 망상.

**경계 규칙**
- 감정: 발생은 사실("화가 올라왔다", "가슴이 답답했다", 몸의 감각 전부) / 감정의 이유·의미는 망상("그 사람 때문에 화가 났다", "화가 난 걸 보니 나는 아직 멀었다"). **한 줄 공식: 감정의 발생은 사실, 감정의 해석은 망상.**
- 말: 실제 발화의 인용은 사실("그가 '알아서 해'라고 말했다") / 발화의 의도·속마음·말투 추정은 망상("귀찮다는 듯이", "무시하는 말투") / 요약·의역된 말("그가 나를 비난했다")은 실제 문장이 없으면 망상 쪽에 놓고 실제 문장이 뭐였는지 되묻는다.
- 방향 중립: 판별은 내용의 긍정·부정과 무관 — 검사는 오직 두 개다. "이번엔 잘될 것 같다"(예언), "그 사람은 나를 아껴준다"(독심), "드디어 운이 풀리기 시작했다"(의미 부여), "나는 원래 위기에 강한 사람이다"(자기 판결) — 칭찬·희망·확신도 카메라에 안 찍히면 전부 망상. **단 긍정 망상 분류 시 교정·경고 톤("그건 착각이에요", "기대하지 마세요") 금지** — 부정 망상과 완전히 같은 건조한 태도로 분류만 한다. 낙관을 꺾는 앱이 아니라 사실과 해석을 구별하는 앱이다.

**내부 감지 범주 (판독용 — 사용자 노출 금지)**
독심(속마음 단정) / 예언(안 일어난 일 단정) / 일반화(항상·절대·또·역시) / 자기 판결(나는 ~한 사람) / 당위(~했어야 했다) / 인과 단정(~때문에 ~됐다) / 의미 부여(~라는 뜻이다). 긍정 문장에도 동일 적용. **범주명을 사용자에게 노출하거나 사용자를 유형으로 명명하는 것은 금지 — 범주는 찾기 위한 도구지 라벨이 아니다.**

**혼합 문장**: 대부분의 문장은 사실과 망상이 섞여 있다. 문장 단위로 분류하지 말고 **성분 단위로 쪼개서** 분류한다.
**판별 불가**: 확신 없는 문장은 억지로 분류하지 않는다. "이 문장은 어느 쪽에 놓으시겠어요?"로 사용자에게 돌려준다. 잘못된 확신보다 되묻기가 항상 낫다.
**대조 출력(6단계)**: 나란히 놓기만. 점수·정답률·맞았다/틀렸다 금지. 사용자가 사실로 놓은 걸 AI가 망상으로 봤으면 "이 문장은 이렇게도 나뉩니다" 형태로만. AI의 구별도 하나의 거울이지 정답이 아니라는 태도 유지.

## 푸시 알림 정책 (지시서 5번) — 두 종류만 존재한다
1. **저녁 질문 푸시 — 매일, 사용자가 정한 고정 시각에 1회.** 오늘의 기록 시간이 열렸음을 알린다. 3일 연속 미기록 상태면 이 푸시의 문구가 바뀐다: "3일째 기록이 없습니다. 안 쓴 날들도 데이터입니다." 별도 추가 발송은 없다 — 문구만 상태를 반영한다
2. **낮 행동 리마인더 — 옵트인.** 10단계에서 사용자가 스위치를 켠 경우에만 발송. 문구는 반드시 사용자 본인이 쓴 행동 문장의 원문 인용: "어제 당신이 정한 행동 — '(원문)'". 앱이 시키는 말투 금지

이 두 개 외의 알림(재촉, 마케팅, 축하)은 만들지 않는다.

## 이 프로젝트는
바이브코딩 밀키트. 모노레포. 재료(통합)는 `packages/integrations/` 에 준비돼 있고, 사용자는 `plan.md` 대로 조립한다.

## 사용자 프로필 (매 세션 자동 로드)
@profile.md
- 프로필이 채워져 있으면 **코딩 수준·글 표현에 눈높이를 맞춰라**: 수준이 낮을수록 비유·용어 풀이를 늘리고 한 번에 한 걸음씩.\n- **비유도 프로필에 맞춰라**: 직업·분야가 있으면 일반 비유 대신 그 업계 일상으로 푼다 (이커머스면 DB=상품 재고 장부, 교육이면 DB=학생부). 같은 개념은 같은 비유로 일관되게.
- **만들고 싶은 것·프로젝트 조건**은 기획과 빌드의 기본 맥락이다. 프로필과 어긋나는 방향(조건에 없는 기능 확장, 스코프 판정 무시)을 잡게 되면 진행 전에 확인부터 하라.

## 사용자에게 말할 때
- 사용자는 **비개발자**일 수 있다. 전문용어(API·DB·배포·커밋 등)는 그 자리에서 한 줄 비유로 풀어라.
- 코드를 보여주기 전, "이게 무슨 일을 하는지" 한국어 1~2줄 먼저.
- "간단합니다"로 뭉개지 말 것.

## 코딩 4원칙 (Karpathy — AI 는 못 짜서가 아니라 브레이크가 없어서 실수한다)
1. **가정하지 말고 물어라.** 요구가 애매하면 멈추고 되묻는다. 조용히 넘겨짚고 밀어붙이지 마라. 해석이 여럿이면 다 제시하고 사용자가 고르게 한다.
2. **최소 코드.** 시킨 문제를 푸는 가장 작은 코드만. 요청 안 한 기능·"나중을 위한" 유연성·안 일어날 상황의 예외 처리 금지.
3. **시킨 것만 고쳐라(수술적).** 인접 코드·주석·포매팅을 "개선"하지 마라. 안 망가진 걸 리팩토링하지 마라. 무관한 데드 코드는 발견해도 건드리지 말고 말만 해라.
4. **검증까지 반복.** "고쳤다"로 끝내지 말고, 실제로 되는지 확인(화면에서 눌러보기·테스트)할 때까지 스스로 돈다. 증거 없이 완료라 말하지 마라.

## 작업 방식
- **먼저 `plan.md` 를 읽어라.** 거기 블럭 순서대로 하나씩 만든다.
- 한 번에 한 스텝만(블럭 안의 체크 한 칸). 끝나면 `plan.md` 체크박스를 갱신한다.
- 막히면 "더 작게 쪼개기". 큰 걸 통째로 시도하지 마라.
- **화면을 만들거나 고칠 땐 그 화면의 `docs/plan/screens/*.md` 를 먼저 읽어라** (있으면). 거기 적힌 **예외 5종(비었을 때·로딩·실패·권한 없음·잘못 입력)까지 처리해야 완성**이다 — 성공 경로만 만들고 끝내지 마라.
- **방향을 정하거나 바꾸는 결정**(기능 빼기·스택 교체·권한/삭제 정책 변경)을 하면 `docs/plan/06-decisions.md` 에 4줄(결정·왜·버린 대안·영향)로 기록한다.
- **다 만든 뒤의 큰 변경은 제안서 먼저.** 새 기능 추가·기존 동작 변경·표 구조 변경·연동 추가라면, 코드를 만지기 전에 `docs/plan/changes/날짜-제목.md` 로 **변경 제안서**를 쓰고 사용자 승인을 받아라:
  ① 뭘 바꾸나 (한 줄) ② 왜 ③ 영향 (어느 화면·표·문·블럭이 바뀌나) ④ 스텝 계획 (체크박스, 확인 방법 포함).
  승인 후 빌드하고, 끝나면 06-decisions.md 에 4줄 요약을 남기고 plan.md 에 블럭·스텝을 추가한다. 제안서 파일은 지우지 않는다 — 그게 변경 이력이다. (오타·문구·한 줄 버그 수정은 제안서 없이 바로.)

## 스택 규칙
- 웹 = `apps/web` (Next.js App Router) · 모바일 = `apps/mobile` (Expo).
- 저장이 필요하면 `packages/integrations/supabase` 를 쓴다. 아주 간단하면 `google-sheets`.
- 키는 항상 `.env` 에서 읽는다. 코드에 키를 박지 마라. Supabase 키 이름은 `SUPABASE_URL` / `SUPABASE_ANON_KEY`.
- **공식 API 가 없는 서비스는 스크래핑·비공식 자동화로 잇지 마라** (네이버 플레이스·카카오톡 개인 채팅 등 — 계정 정지·법적 위험). blueprint 의 공식 문 판정(7-6a)을 따르고, 미개방이면 대안(수동 입력·공식 채널·다른 서비스)을 제시한다.
- **테이블 만들기·바꾸기(SQL)는 CLI 없이 간다.** AI 가 SQL 전문을 만들어 보여주면, **실행은 사용자가 Supabase 대시보드 → SQL Editor 에 붙여넣어서** 한다("창고에 선반 놓기"로 안내). Supabase CLI·Vercel CLI 설치를 권하지 마라 — 이 킷의 기본 경로는 웹 콘솔이다. 실행 후 Table Editor 에서 표가 생겼는지 사용자가 눈으로 확인하게 하라.
- **`NEXT_PUBLIC_` 접두사는 이 킷에서 금지.** 환경변수를 새로 만들 때도 붙이지 마라. 브라우저(화면 코드)에서 읽어야 하는 공개 키는 `apps/web/next.config.mjs` 의 `env` 목록에 추가하는 방식으로만 처리한다.
- 비교는 `===` / `!==` 만. `==` / `!=` 금지. null 은 명시적으로 확인.
- DB 관계는 **꼬리표 컬럼으로만, FOREIGN KEY 제약은 걸지 않는다** (정합성은 soft delete + 앱 로직). FK 제약이 없으면 Supabase 중첩 조회가 안 되니 **조인은 두 번 나눠 읽거나 뷰로**.
- **N+1 금지**: 목록을 그릴 때 항목마다 창고를 또 부르지 마라 — 필요한 걸 한 번에 모아 읽는다(`in` 조회·뷰). 검색·정렬·걸쇠 컬럼엔 인덱스를 걸어라. 둘 다 사용자에게 물을 일이 아니라 네가 챙길 일이다.

## 보안 — 치명적인 것만 자동으로 막는다
> 원칙: 개발·디버깅을 방해하지 않는다. 아래 **치명 3개**(되돌릴 수 없는 사고)만 사용자가 안 시켜도 자동 적용하고, 나머지(RLS·입력검증·XSS)는 개발 중엔 강제하지 않는다 — 공개 배포 직전에 `security-reviewer` 로 점검한다.

**치명 1 — 시크릿 노출** (털리면 끝)
- 키는 `.env` 에만. 코드·깃허브·로그에 절대 노출 금지(`.gitignore` 에 `.env` 확인).
- `SUPABASE_SERVICE_ROLE_KEY` 같은 마스터 키(RLS 우회)는 **절대 클라이언트에 두지 마라** — next.config 의 env 공개 목록에도 올리지 마라. 서버에서만. (반면 `SUPABASE_ANON_KEY` 는 공개 전제 키라 노출돼도 됨.)

**치명 2 — 남의 데이터 파괴** (복구 불가)
- 사용자 데이터 테이블에 **공개(anon) 삭제·수정 정책을 열지 마라.** 누구든 남의 데이터를 지우는 사고. (읽기 공개는 개발 중 허용 — 이게 디버깅을 편하게 한다.)
- 삭제는 되도록 soft delete(`deleted_at` 표시). 실서비스 DB 대량 삭제·TRUNCATE 는 사용자 확인 없이 금지.

**치명 3 — 결제·인증 사고** (돈·계정)
- 결제·인증은 먼저 **테스트 키/테스트 모드**로 만든다. 실제 키 전환은 맨 마지막.

**배포 전 점검 (개발 중엔 강제 안 함 — 디버깅 우선)**
- **RLS(남의 데이터 열람 차단)**: 개발 중엔 꺼둬도 된다(그래야 데이터가 안 보이는 디버깅 지옥을 피한다). 단 이 앱을 **남에게 공개**하기 전엔 켜야 한다 — 공개 배포 직전 `security-reviewer` 가 "이거 켜세요" 하고 짚는다.
- 입력 검증·XSS 도 배포 전 점검 항목. 개발 중엔 기능부터, 공개 전에 리뷰어가 훑는다.
- 즉 **혼자 쓰는 데모·개발 단계 = 편하게, 남에게 공개 = security-reviewer 통과 후.**

## 안전
- `.env` 를 읽거나 출력하지 마라.
- 파괴적 명령(rm -rf, DB 삭제·TRUNCATE·대량 UPDATE)은 사용자 확인 없이 실행 금지. 특히 실서비스 DB.
