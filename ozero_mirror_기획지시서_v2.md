# 오제로의 거울 — 기획 지시서 v2

이 문서는 v1을 대체한다. v1과 충돌하면 이 문서가 우선한다.

## 0. 지금 단계에 대한 지시

너는 지금 코드를 작성하지 않는다. 이 문서를 전부 읽고, 아래 기획 산출물을 먼저 만든다.

1. PRD (제품 요구사항 문서) — 이 지시서를 구조화하고 빈 곳을 드러낸 것
2. 화면 목록과 각 화면의 요소 설명 (와이어프레임 수준의 텍스트 설명)
3. DB 스키마 초안 (Supabase 기준, 테이블·관계·RLS 정책·아래 7번의 구조화 저장 원칙 반영)
4. AI 프롬프트 설계 초안 (아래 9번의 5개 AI 동작 각각에 대해, 실패 사례 포함)
5. 미결정 질문 목록 — 이 문서에서 판단이 불가능한 항목은 추측하지 말고 전부 여기에 올린다

모호한 부분을 임의로 채우는 것이 가장 나쁜 결과다. 모르면 질문 목록에 넣어라.

## 1. 제품 정의

21일간 매일 한 번, 사용자가 자기 하루를 기록하고 그 기록에서 사실과 자동해석(이하 "망상")을 스스로 구별하는 훈련 웹앱. 목표 상태는 알아차림→선택→행동. 감정 관리 앱도, 위로 앱도, 일기 앱도 아니다. 부정적 생각을 긍정적 생각으로 바꾸는 앱은 더더욱 아니다 — 해석과 사실을 분리하는 근육을 만드는 훈련 도구다.

이 앱의 본질은 챗봇이 아니라 판독과 회수다: 사용자의 기록에서 사실·망상·감정·행동을 분리해 저장하고, 사용자의 과거 문장을 원문 그대로 되돌려주는 것.

첫 사용자는 신규 대중이 아니라 "오제로 퀘스천 레터" 수료자다. 방법론을 이미 아는 사람들이 먼저 쓴다.

## 2. 절대 원칙 — 위반 불가

1. **AI는 거울이지 분석가·선생·상담사가 아니다.** 채점하지 않고, 정답을 주지 않고, 진단하지 않고, 사용자를 유형이나 성격으로 명명하지 않는다. AI의 출력은 항상 "대조"와 "회수"다.
2. **사용자가 쓰지 않은 것을 지어내지 않는다.** AI가 과거 기록을 참조할 때는 반드시 원문 인용 + 작성 날짜 형태로만 한다. 요약·의역·패턴 명명 금지. 인용할 문장이 없으면 인용하지 않는다.
3. **톤 규칙:** 위로 언어 금지. 이모지 금지. 감탄·과잉 반응 금지. 극적인 비유 금지. 판단·충고·조언·평가 금지. 영어 외래어 최소화. 짧은 단정문. ~이에요/~해요체. 건조하고 중립적인 거울의 말투.
4. **추상 언어의 구체화는 바꿔주기가 아니라 되묻기로 한다.** 사용자가 "힘들었다"고 쓰면 AI가 "지쳤다"로 바꾸지 않는다. "'힘들었다'고 쓰셨어요. 오늘 어떤 장면에서였나요"처럼 사용자가 스스로 구체화할 자리를 만든다.
5. **행동의 주어는 사용자다.** 하루의 마지막 단계에서 작은 행동을 정하는 것은 사용자이며, AI 제안은 사용자가 명시적으로 요청할 때만 노출한다.
6. **사용자 간 데이터 격리는 브랜드 생사 문제다.** 격리(데이터베이스 층)와 공유(제품 층)를 구분한다. 격리는 예외 없이 적용하고, 공유는 사용자의 건별 선택으로만 일어난다. 상세는 6번 항목.

## 3. 하루 루프 — 10단계

순서 변경 금지. 특히 5번(사용자 구별)이 6번(AI 대조)보다 먼저인 것이 이 앱의 핵심이다. AI가 먼저 답을 주면 훈련이 아니라 의존이 된다. 타인의 내용은 어떤 형태로든 제출 전에 노출되지 않는다 — 오늘 기록이 타인의 글에 물들면 자기 기록이 아니게 된다.

1. **오프닝 — 어제의 회수** — 현존 감각을 깨우는 진입 화면. 내용물은 두 가지: (a) 어제 사용자가 쓴 문장 중 하나를 날짜와 함께 원문 인용 (분석·해석 금지, 인용만), (b) 어제 정한 작은 행동의 실행 여부 표시 (했다/안 했다/부분. 평가 문구 없음). 어제 기록이 없으면 인용하지 않는다.
2. **오늘 상태 체크** — 기분·감정·체력·수면을 숫자로. 10초 안에 끝나는 UI
3. **오늘의 기록** — 자유 기록
4. **(시스템) 위기 신호 분기** — 기록에 자해·자살·타해 신호가 감지되면 이후 단계로 진행하지 않고 전문 지원 안내 화면으로 전환. 감지 방식과 안내 문구는 미결정 질문으로 올릴 것
5. **사실·망상 구별** — 사용자가 자기 기록에서 직접 구별
6. **AI 거울 대조** — AI가 같은 기록에서 구별한 결과를 사용자의 구별과 나란히 표시. 채점 아님. "당신은 이렇게 나눴고, 이렇게도 나뉩니다" 구조
7. **망상·감정 연결** — 분리된 망상과 2번에서 기록한 감정 숫자를 나란히 놓고, 어떤 망상이 어떤 감정을 만들었는지 사용자가 잇는다
8. **오늘의 질문** — AI가 생성. 반드시 누적 기록 기반. 개인화의 방법은 요약이 아니라 원문 인용: "6월 12일에 당신은 이렇게 썼습니다 — '(원문)'. 오늘의 기록과 나란히 놓아보세요" 형태
9. **답변** — 사용자 작성. 제출 시 "이 답변을 다른 관찰자에게 익명으로 보여줄까요" 건별 토글 (상세는 6번)
10. **작은 행동 선택** — 사용자가 내일 할 작은 행동을 스스로 정한다. "AI 제안 보기"는 버튼 뒤에 숨긴다. 이 화면에서 "낮에 이 행동을 알림으로 받을까요" 스위치를 사용자가 켤 수 있다 (기본: 꺼짐)

제출 후 이전 단계로 돌아갈 수 없다. 내용 수정은 불가하되, 오타 수정 허용 여부·범위는 정책 옵션으로 제안할 것.

## 4. 21일 구조

- **개인화 시작.** 기수제 아님. 각 사용자는 자기 시작일부터 자기 21일을 걷는다
- **하루 1회 활성화 + 사용자 지정 고정 시각.** 가입 시 사용자가 자기 시각을 하나 정한다(기본 제안: 저녁). 24시간 게이팅은 반드시 서버 사이드에서 판정 (클라이언트 판정 금지)
- **7일째·14일째 중간 거울.** 분석 보고서가 아니라 짧은 사실 제시. 예: "지금까지 기록에서 가장 많이 반복된 단어 세 개." 해석 없이 사실만
- **14일째 시차 대면.** 14일째 화면에서 1일째 답변 전문을 그대로 보여준다
- **빠진 날 처리 — 가장 중요한 설계.** 빠진 날은 실패가 아니라 데이터다. 복귀한 날 오프닝: "어제는 기록이 없는 날입니다. 안 쓴 날은 어떤 날이었나요." 연속 기록이 깨졌다는 표시·문구 일절 금지
- **21일 보고서.** 21일치 기록에서 패턴을 원문 인용 중심으로 제시. 형식은 기획 산출물에서 제안할 것
- **21일 격자 달력.** 기록된 날이 채워지는 화면. 배지·포인트·불꽃·연속기록 카운터는 절대 금지 — 사실의 표시만

## 5. 푸시 알림 정책

두 종류만 존재한다.

1. **저녁 질문 푸시 — 매일, 사용자가 정한 고정 시각에 1회.** 오늘의 기록 시간이 열렸음을 알린다. 3일 연속 미기록 상태면 이 푸시의 문구가 바뀐다: "3일째 기록이 없습니다. 안 쓴 날들도 데이터입니다." 별도 추가 발송은 없다 — 문구만 상태를 반영한다
2. **낮 행동 리마인더 — 옵트인.** 10단계에서 사용자가 스위치를 켠 경우에만 발송. 문구는 반드시 사용자 본인이 쓴 행동 문장의 원문 인용: "어제 당신이 정한 행동 — '(원문)'". 앱이 시키는 말투 금지

이 두 개 외의 알림(재촉, 마케팅, 축하)은 만들지 않는다.

## 6. 공유와 격리 — 두 층의 분리

**격리 층 (예외 없음):** 자유 기록(3단계), 사실·망상 구별(5단계), 상태 체크 숫자는 어떤 경우에도 본인과 관리자 외에 노출되지 않는다. RLS로 강제한다.

**공유 층 (건별 선택):** 공유 가능한 것은 오늘의 질문에 대한 답변(9단계) 하나뿐이다. 매일 답변 제출 화면에서 그 답변 하나에 대해 토글로 묻는다. 기본값: 비공개. 전체 설정의 일괄 공개 옵션은 만들지 않는다.

**동료 존재감 장치 (커뮤니티 기능 없이):**
1. 기록 제출 직후 한 줄: "오늘 N명의 관찰자가 함께 기록했습니다" (달력 날짜 기준 카운트)
2. 오늘 기록을 제출한 뒤에만, 다른 관찰자의 공개 동의된 익명 답변 1개가 열린다. 제출 전 노출 금지
3. 사용자는 익명 관찰자 코드로 표시된다 (예: o024. 가입 순번 기반). 코드는 이름을 가리지만 내용의 식별 가능성까지 가리지 못하므로, 공유 범위가 답변 하나로 한정되는 위 규칙이 함께 있어야 한다

댓글·피드·채팅·팔로우는 만들지 않는다.

## 7. 데이터·보안·법 — 구조화 저장이 이 앱의 해자다

- **구조화 저장 원칙.** 모든 기록은 한 덩어리 텍스트가 아니라 단계별 별도 필드로 저장한다: 상태 숫자 / 자유 기록 / 사용자의 사실·망상 구별 / AI의 대조 결과 / 망상·감정 연결 / 오늘의 질문(인용된 원문의 출처 날짜 포함) / 답변 / 선택한 행동 / 다음날 실행 여부. 이 구조가 나중에 패턴 지도를 만드는 원재료다. 스키마 설계 시점에 정해지며 나중에 바꿀 수 없다고 간주하라
- **동의 3종 필드를 스키마에 처음부터:** (a) 심리 기록의 AI 전송·분석 동의, (b) 답변의 익명 공개 동의(건별), (c) 익명화된 기록의 서비스 개선·자산화 활용 동의. 각 동의의 시각과 버전을 기록한다
- **전후 비교 조회 구조.** 1일·7일·14일·21일 답변이 날짜 태깅으로 비교 조회 가능해야 한다 (14일 시차 대면과 21일 보고서, 향후 변화 증거 축적의 기반)
- Supabase. 사용자 소유 테이블 전체에 RLS를 처음부터 적용
- 관리자(운영자 1인)는 전체 기록 열람 대시보드 필요 — 열람 사실이 약관·동의에 명시되는 구조로
- 사용자 삭제 요청 시 데이터 완전 삭제 흐름 포함
- 개인정보처리방침·이용약관에 들어가야 할 항목 목록을 기획 산출물에 포함 (법률 자문 전 초안 수준)
- 측정을 처음부터 로깅: 며칠째에 이탈하는가(날짜별), 하루 루프의 어느 단계에서 이탈하는가(단계별), 답변 길이 추세

## 8. 기술 제약

- **모든 AI 프롬프트는 서버 사이드(Edge Functions)에만 존재한다.** 클라이언트 코드·네트워크 응답에 프롬프트가 노출되지 않게 한다
- 다국어(i18n) 구조를 처음부터. 한국어 우선, 번역 파일 분리 방식. 단 AI 출력의 톤 규칙은 한국어 기준으로 먼저 완성
- 결제는 마지막 빌드 단계. 초기 사용자는 결제 없이 관리자 수동 등록으로 운영 가능해야 한다
- localStorage 의존 금지, 모든 상태는 서버에
- 모바일 우선. 사용자 대부분이 스마트폰으로 밤에 사용

## 9. AI 동작 5개 — 프롬프트 설계 대상

각각에 대해 시스템 프롬프트 초안과 실패 사례(하면 안 되는 출력 예시)를 함께 작성할 것.

1. 사실·망상 대조 (6단계)
2. 오늘의 질문 생성 — 누적 기록에서 원문 인용 회수 (8단계)
3. 중간 거울 — 반복 단어 등 사실 제시 (7·14일)
4. 21일 보고서
5. 위기 신호 감지 (4단계)

공통 품질 기준 두 개: 정확한가, 그리고 호들갑이 없는가. 두 번째가 첫 번째만큼 중요하다.

## 10. 빌드 순서 (기획에 반영할 단계 구분)

1. 코어 루프 (하루 10단계, 구조화 저장, 본인 열람)
2. 인증 + RLS + 동의 3종 + 관리자 대시보드
3. 누적 질문·시차 대면·서버 게이팅·중간 거울·21일 보고서·푸시 2종
4. 동료 존재감 (N명 카운트, 익명 답변 1개, 건별 토글)
5. 결제 (건당·구독)

## 11. 하지 말 것

- 스트릭 배지, 포인트, 불꽃, 레벨, 축하 애니메이션
- 위로하는 챗봇 톤, 사람인 척하는 알림
- 부정적 해석을 긍정으로 바꾸라는 유도
- **AI가 사용자를 유형·성격·패턴명으로 자동 분류하는 것 — v1 범위 밖.** 패턴 분류 체계는 데이터가 쌓인 뒤 운영자가 오프라인에서 만드는 자산이지, 앱이 사용자 면전에서 붙이는 라벨이 아니다
- 파인튜닝, 벡터 DB 유사 사용자 매칭 — v1 범위 밖
- 제출 전 타인 콘텐츠 노출
